  • Initial Post

    This paper talked about users' participation in information security risk management (ISRM) and how it may contribute to efficient risk management in organizations. This paper made use of a multi-method approach that involves a combination of qualitative and quantitative methods to examine user participation in SRM. From the study, qualitative methods were used to provide an understanding of the activities, behaviors, and assignments that define user participation which led to the development of a theoretical model. This theoretical model developed was then tested using the quantitative method which led to a richer and more comprehensive understanding of the study (Spears & Barki, 2010).

    Users are generally presented as the weakest link in security in typical information security literature, but this study highlighted the benefits of involving users in the risk management process. Users participating in SRM led to an increased organization awareness of information security risks due to users being assigned SRM activities including working on specific controls giving them more visibility to security risks and controls for mitigating these risks. User participation in SRM contributed to improvements in control design due to the valuable feedback that was received from them to confirm if the controls were feasible to be used on a day-to-day basis. (Spears & Barki, 2010)

    By having users participate in SRM, users that are considered the weakest link in information security became the greatest asset in the effort to make reduce risk related to information security and make security better aligned with business objectives Deli et al. (2017).

    Users being able to be assigned responsibilities and activities is key to analysis in SRM. Without user access, the qualitative method, which is meant to be exploratory, relatively easy, cheap, and not based on any previous theories, concepts will be used in a limited way. The quantitative method will have to be explored which typically deals with the complete investigation meaning that analysis will generally be more expensive, will demand more detailed tools (Rot, 2008), or else it will be performed using generic information and guesses.

    References

    2. Deli, M., Ahmad, J., Hassan, N., Maaroop, N., Narayana, S., Abdullah, M. & Yacoob, S. (2017) Understanding User Participation in Information Security Risk Management Available from: http://eprints.utm.my/id/eprint/80684/1/NoorHafizahHassan2017_UnderstandingUserParticipationinInformation.pdf [Accessed 21 June 2022].

    Abidemi’s Preparation on the overall risks and Mitigations
    Phase of Software Development Life cycle (SDLC) Risk Factor Mitigation Technique
    Risks common to all SDLC phases Continuous Requirement Changes
    1. Stabilizing requirements and specifications as early as possible.
    2. User Surveys
    3. Develop prototyping and have the requirements reviewed by the client.
    4. Utilizing Change Control Board
    Requirement Analysis and Planning Phase Unrealistic schedules and Budget estimations
    1. Requirement Scrubbing.
    2. Multisource cost and Schedule Estimation
    Requirement Analysis and Planning Phase Gold plating/scope creep
    1. Separating project into manageable segments
    2. Designing to cost.
    3. Prototyping
    4. Requirement scrubbing
    Development and Implementation Developing the wrong Software functions
    1. Prototyping
    2. Provide scenarios methods
    3. User Participation.
    4. Task Analysis.
    Operation and Maintenance Phase. User Resistance to Change.
    1. Educating users on the impact of changes during the software project.
    2. Provide training in the new technology and organize domain knowledge training.
    3. Participating users during the entire software project lifecycle.
    Teams 5 Overall Causes of Risks
    1. Poor project/change management.
    2. Conflicting/unclear/ever changing requirements.
    3. Technical risks, e.g. hardware, incorrect implementation, incorrect integration between different systems, incorrect architecture, wrong programming language, etc.
    4. Human risks, e.g. users resisting changes, lack of buy-in, etc.
    5. Organizational/Business risks, e.g. software design doesn’t meet business requirements, software implementation doesn’t meet business processes, etc.
    Teams 5 Possible Mitigations
    1. Ensure that the project is managed by an experienced Project Manager.
    2. Review and document all software requirements, changes and utilize a framework.
    3. Ensure that the security is built into the code or project, from conception to the delivery to apply (Secure SDLC).
    4. See user’s involvement in the software development and implementation, this proved to be of value to successful project delivery.
    5. Anticipate possible risk factors in each phase e.g risks outside of the control of the organization such as political, unrest, tax and inflation.
